# Transparent Proxy in Kubernetes (Sidecar Mode)

ClawGuard can act as a fully transparent L7 proxy for egress traffic when injected as a sidecar into a Kubernetes Pod.
In this mode, your application containers do not need to configure `HTTP_PROXY` or `HTTPS_PROXY` environment variables, nor do they need to support proxies at all. ClawGuard intercepts the outbound traffic seamlessly at the network layer.

## Architecture

To make the proxy "transparent", we apply three primary mechanisms:

1. **iptables (Init Container):** An init container with `NET_ADMIN` privileges configures the pod's internal network namespace to redirect outbound TCP traffic on ports 80 and 443 to ClawGuard's listener ports (e.g., 8080 and 8443).
2. **UID Exclusion:** The `iptables` rules must ignore traffic originating from ClawGuard's own user ID, preventing an infinite routing loop.
3. **SNI Routing and Cert-Manager:** For HTTPS, ClawGuard relies on Node's `SNICallback` to intercept the TLS handshake and forge a certificate on the fly. To ensure the client container trusts these certificates, we use Kubernetes [cert-manager](https://cert-manager.io) to generate a Certificate Authority (CA). The secret generated by cert-manager is then mounted:
   - Into ClawGuard (so it has `tls.crt` and `tls.key` to forge certificates).
   - Into the client container (so the client OS trusts the CA).

## How to use

Ensure `transparentProxy.enabled` is set to `true` in your `clawguard.yaml`:

```yaml
transparentProxy:
  enabled: true
  httpPort: 8080
  httpsPort: 8443
```

Check out the full example provided in `examples/k8s-transparent-proxy.yaml`. This example demonstrates:
- A `cert-manager` self-signed Issuer and CA Certificate.
- A `ConfigMap` storing `clawguard.yaml`.
- A Pod with an `iptables` Init Container, the `clawguard` sidecar running as UID 1000, and a `client` container equipped with `curl` for testing.

### Testing Transparent Proxy Flow

If you deploy the example Pod, you can shell into the client container and execute requests normally. Because of `iptables`, the network routes the packets to ClawGuard:

```bash
kubectl exec -it client-with-clawguard -c client -- sh
# Standard HTTP
curl -i http://api.openai.com/v1/models

# Standard HTTPS (Client trusts the injected CA)
curl -i https://api.openai.com/v1/models
```

You can view ClawGuard's logs in the sidecar to verify the interception:
```bash
kubectl logs client-with-clawguard -c clawguard
```
